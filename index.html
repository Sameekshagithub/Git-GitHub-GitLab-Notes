Git

Distributed Version Control System — created by Linus Torvalds, 2005

What is Git?

Distributed VCS Core

Every developer has a full copy of the repository including the entire history. No single point of failure.

Snapshots, Not Diffs

Git stores data as snapshots of the project over time, not as a list of file-based changes.

Integrity

Everything is checksummed with SHA-1 before storing. You can't change content without Git knowing.

The Three States

Modified → changed but not committed. Staged → marked for next commit. Committed → safely stored in local DB.

Branching & Merging

💡

Branches are cheap in Git — they're just 41-byte pointers. Create them often and keep your work isolated.

CommandDescription
git branchList local branches
git branch -aList all branches including remotes
git branch <name>Create a new branch
git switch <name>Switch to an existing branch
git switch -c <name>Create and switch to new branch
git merge <branch>Merge specified branch into current branch
git merge --no-ff <branch>Merge preserving branch history (no fast-forward)
git branch -d <name>Delete a merged branch
git branch -D <name>Force delete a branch (even if unmerged)

Merge Strategies

Fast-Forward

Moves the branch pointer forward. No merge commit created. History is linear. Used when target is directly ahead.

3-Way Merge

Creates a new merge commit when branches have diverged. Preserves context of both branches.

Squash Merge

Combines all commits from feature branch into a single commit. Cleaner main history but loses individual commit detail.

Rebasing

⚠️

Golden rule: Never rebase commits that have been pushed to a shared remote branch — it rewrites history and causes problems for others.

CommandDescription
git rebase <base>Reapply commits on top of another branch
git rebase -i HEAD~3Interactive rebase for last 3 commits (squash, reword, drop…)
git rebase --continueContinue rebase after resolving conflicts
git rebase --abortCancel the rebase and return to original state

GitHub

Web-based hosting for Git repositories — owned by Microsoft since 2018

Platform Overview

Repositories Core

Public or private storage for your code with full Git history, issues, wikis, and project boards.

Pull Requests

The primary collaboration mechanism — propose, review, discuss, and merge changes with inline comments and review assignments.

GitHub Actions

Built-in CI/CD automation. Define workflows in YAML to build, test, and deploy on every push or PR.

GitHub Pages

Free static site hosting directly from a repository. Supports Jekyll themes and custom domains.

GitHub Copilot

AI pair programmer powered by OpenAI Codex. Suggests code completions in real-time inside your editor.

GitHub Packages

Host and manage packages (npm, Docker, Maven, NuGet, RubyGems) alongside your source code.

GitHub Actions

Workflow

Automated process defined in a .yml file inside .github/workflows/. Triggered by events.

Jobs

A workflow contains one or more jobs. Jobs run in parallel by default, or sequentially with needs:.

Steps

Individual tasks in a job. Steps can run shell commands or use marketplace Actions.

Runners

Virtual machines that execute jobs. GitHub-hosted (Ubuntu, macOS, Windows) or self-hosted.

# .github/workflows/ci.yml — example CI workflow name: CI Pipeline on: push: branches: [ main, develop ] pull_request: branches: [ main ] jobs: build-and-test: runs-on: ubuntu-latest steps: - name: Checkout code uses: actions/checkout@v4 - name: Set up Node.js uses: actions/setup-node@v4 with: node-version: '20' - name: Install dependencies run: npm ci - name: Run tests run: npm test

Security Features

  • Dependabot — automatic pull requests to update vulnerable dependencies.
  • Secret scanning — detects accidentally committed API keys and tokens.
  • Code scanning (CodeQL) — static analysis to find security vulnerabilities in code.
  • Branch protection rules — require reviews, passing CI, and up-to-date branches before merging.
  • CODEOWNERS — file that auto-assigns reviewers based on which files are changed.
  • Two-factor authentication (2FA) — required for contributors on certain plans/orgs.

GitLab

Complete DevSecOps platform — available as cloud SaaS and self-hosted

Platform Overview

GitLab CI/CD Built-in

First-class CI/CD with .gitlab-ci.yml. Pipelines, stages, jobs, and environments all defined as code.

Merge Requests

GitLab's equivalent of Pull Requests — with draft MRs, approval rules, merge trains, and squash options.

GitLab Registry

Built-in Docker container registry and package registry for npm, Maven, PyPI, etc.

GitLab Pages

Deploy static websites from CI/CD pipelines with custom domains and Let's Encrypt certificates.

Security & Compliance

SAST, DAST, dependency scanning, container scanning, license compliance — all integrated in the pipeline.

Self-Hosted Option

Run GitLab on your own servers (Community Edition free, Enterprise Edition paid). Full control over data.

GitLab CI/CD Pipeline

✅

GitLab CI/CD is configured via a single .gitlab-ci.yml file in the root of the repository. GitLab automatically detects it on every push.

# .gitlab-ci.yml — example pipeline stages: - build - test - deploy variables: NODE_ENV: production build-job: stage: build image: node:20 script: - npm ci - npm run build artifacts: paths: [ dist/ ] test-job: stage: test image: node:20 script: - npm test coverage: '/Coverage: \d+\.\d+/' deploy-prod: stage: deploy script: - echo "Deploying to production…" environment: name: production url: https://myapp.example.com only: - main
ConceptDescription
StagesSequential groups. All jobs in a stage run in parallel before moving to next stage.
JobsIndividual units of work. Each runs in a clean runner environment.
RunnersAgents that execute jobs. Shared (GitLab.com), group-level, or project-specific.
ArtifactsFiles produced by a job, passed to subsequent stages or downloadable.
CacheSpeed up pipelines by caching node_modules, .pip etc. between runs.
EnvironmentsTrack deployments to named envs (staging, production) with rollback support.
include:Reuse CI/CD configuration from other YAML files or templates.
rules: / only:Conditionally control when a job runs (branch, tag, MR, schedule…).

Merge Requests (MR)

  • Draft MR — prefix title with Draft: to signal work in progress; blocks accidental merges.
  • Approval rules — require 1+ approvals from specific users or groups before merge.
  • Merge trains — queue MRs to be merged one-by-one maintaining CI green state.
  • Auto-merge — automatically merges when all conditions (CI, approvals) are satisfied.
  • Related issues — use Closes #123 in description to auto-close issues on merge.
  • Reviewers — assign dedicated reviewers separate from code owners.

Groups, Subgroups & Namespaces

Groups

Organise related projects together. Manage members, CI/CD variables, and runners at the group level.

Subgroups

Nest groups up to 20 levels deep. Inherit members and settings from parent groups.

Roles

Guest, Reporter, Developer, Maintainer, Owner — each with progressively more permissions.

Git Workflows

Industry-standard branching strategies for teams

Git Flow

💡

Best for projects with scheduled releases. More complex but very structured.

main

Production-ready code. Only receives merges from hotfix and release branches. Tagged with version numbers.

develop

Integration branch. All feature branches merge here. Represents latest delivered development changes.

feature/*

Branched off develop. One branch per feature. Merged back to develop when complete.

release/*

Branched off develop for release preparation. Only bug fixes, merges to both main and develop.

hotfix/*

Emergency patches branched from main. Merged back to both main and develop immediately.

GitHub Flow

✅

Simple and effective for continuous deployment. Used by GitHub themselves. main is always deployable.

branch from main
→
commit changes
→
open PR
→
discuss & review
→
deploy & test
→
merge to main

Trunk-Based Development

  • All developers commit directly to main (trunk) or use very short-lived feature branches (max 1–2 days).
  • Feature flags used to hide incomplete features in production.
  • Requires a strong CI/CD pipeline and high test coverage.
  • Enables true continuous integration — no long-lived branches means no merge hell.
  • Used by Google, Facebook, and many high-performing engineering teams.

Forking Workflow

  • Each contributor forks the official "upstream" repository to their own account.
  • Changes are pushed to the personal fork, then a PR is opened to the upstream.
  • Maintainers never give write access to contributors — ideal for open-source projects.
  • Upstream changes are synced: git remote add upstream <url> → git fetch upstream → git merge upstream/main

When to Choose

Choose GitHub if…

You want the largest developer community, best open-source collaboration, GitHub Copilot integration, and Microsoft ecosystem tools.

Choose GitLab if…

You need self-hosting, a single platform for the full DevSecOps lifecycle, enterprise compliance, or advanced CI/CD without third-party tools.

Both platforms…

Use Git underneath. You can push the same repository to GitHub and GitLab simultaneously using multiple remotes.
